# PR Review — 2.5D Sprites (`lite-sprites-pr3`)

| Priority    | Issue                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              | State |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----- |
| 🔴 High     | `depth: "test-write"` does not actually write depth — `sprite-pipeline.ts` always sets `depthWriteEnabled: false`. Both `"test"` and `"test-write"` resolve to identical depth state (only the renderable bucket differs), despite docs promising `depthWrite: true`. Need to thread depth-write through `getOrCreateSpritePipeline`, include it in `spritePipelineKey`, and have `buildSpriteRenderable` pass `layer.depth === "test-write"`.                                                                                     | NEW   |
| 🔴 High     | Opaque depth-hosted sprite renderables can be baked into the scene opaque render bundle with stale state. `scene-core.ts` only rebuilds the opaque bundle when `_renderableVersion` / `_vis` changes, but `Sprite2DLayer` add/remove/grow/count changes do not bump the scene version. After first render, a `depth: "test-write"` layer can keep drawing an old instance count or old instance buffer. Mirror `SpriteRenderer`'s bundle invalidation strategy or keep dynamic sprite renderables out of the cached opaque bundle. | NEW   |
| 🔴 High     | Scene 52/53 parity specs are Lite-self regression tests, not Babylon.js parity tests. They skip when `reference/scene52-hud-on-3d/babylon-ref-golden.png` / `reference/scene53-depth-hosted-sprites/babylon-ref-golden.png` are missing, and this branch does not include those reference dirs or `lab/public/thumbnails/scene52.png` / `scene53.png`. This violates `GUIDANCE.md` scene-addition/reference conventions and can let `pnpm test:parity` pass without validating these new scenes.                                   | NEW   |
| 🔴 High     | Bundle ceilings for scenes 52/53 are placeholders (`maxRawKB: 200`) — roughly 2× actual baseline. Defeats the regression-detection purpose. Measure real KB and set ~5 KB above (matches all other scenes in `scene-config.json`).                                                                                                                                                                                                                                                                                                 | NEW   |
| 🟡 Medium   | Per-renderable pipeline cache duplication: every depth-hosted layer constructs its own `SpritePipelineCache` in `sprite-renderable.ts`, so two layers in one scene = two `GPUShaderModule` + duplicate pipelines for the same key. `SpriteRenderer` correctly shares one cache; renderable path should too (lazy-init on engine or `SceneContextInternal`).                                                                                                                                                                        | NEW   |
